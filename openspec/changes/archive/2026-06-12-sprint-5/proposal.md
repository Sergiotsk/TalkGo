# Sprint 5 Proposal — Alpha con Usuarios Finales

## Intent

Deploy TalkGo to a real VPS so that 5 non-technical users can test bidirectional voice translation over the internet. This sprint replaces every development-time stub (PassthroughCodec, STUN-only ICE, plain HTTP) with production-grade equivalents and adds the minimal operational guardrails (rate limiting, error UX, feedback) needed to avoid burning the alpha budget.

## Scope

### In Scope

- **Real Opus codec** — Replace `PassthroughCodec` with a real Opus decode/encode adapter so the translation pipeline actually converts audio formats. Use `pion/opus` (pure Go, no CGO) as the implementation. This is the ONE allowed new Go dependency.
- **TURN server integration** — Extend `webrtc.Config` to accept TURN credentials and inject them into ICE configuration. Server-side only; Coturn itself is deployed via Docker Compose (ops artifact, not Go code).
- **HTTPS/WSS termination** — Caddy reverse proxy handles TLS termination and auto-cert via Let's Encrypt. The Go server continues to listen on plain HTTP behind Caddy. Zero Go code changes for TLS.
- **Docker Compose deployment** — Single `docker-compose.yml` with three services: `talkgo` (Go server), `coturn` (TURN), `caddy` (reverse proxy + HTTPS). Dockerfile for the Go binary (multi-stage build).
- **Rate limiting** — Server-side token bucket per IP address, stdlib-only (`sync.Mutex` + `time.Ticker`). Applied as HTTP middleware on `POST /rooms` and `GET /ws/{roomID}`. Configurable via environment variables.
- **Error UX** — Structured error events over the existing WebSocket signaling channel. Server sends typed error messages (`ice-failed`, `server-error`, `rate-limited`, `peer-timeout`, `translation-error`) that the client can render. No new transport — extends the current `EventNotifier` / `NotifySession` mechanism.
- **Feedback endpoint** — `POST /feedback` that accepts a JSON body `{session_id, rating, comment}` and appends to a structured log (slog). No database, no new dependencies.
- **Expo Go distribution** — Mobile alpha testing via Expo Go. Testers install the Expo Go app (App Store / Play Store), scan a QR code generated by `npx expo start`, and connect to the production server via WSS. Requires: (a) the React Native client configured with the production server URL, (b) HTTPS/WSS already in scope, (c) a tester guide in `docs/deploy/`. No server-side code changes — purely a client + ops concern.
- **Environment configuration** — Move hardcoded values (listen address, TURN credentials, OpenAI API key, rate limits) to environment variables. Read via `os.Getenv` with sensible defaults. No config library.
- **Deployment documentation** — `docs/deploy/` with setup instructions for the VPS, Docker Compose, DNS, and Coturn configuration.
- **Health check hardening** — Extend `GET /health` to report TURN reachability and OpenAI API key presence (not the key itself).

### Out of Scope

- **Mobile client distribution via stores** — TestFlight, APK signing, or Play Store. Expo Go covers the alpha distribution entirely.
- **Expo builds (EAS Build)** — not needed for 5-user alpha. Expo Go dev client is sufficient.
- **Expo builds (EAS Build / standalone binaries)** — not needed; testers use Expo Go directly.
- **Horizontal scaling** — Single VPS, single process. No load balancer, no Redis, no distributed state.
- **Persistent storage** — `InMemoryRoomRepository` stays. At 5 users, durability is not needed.
- **Authentication/authorization** — No user accounts, no JWT, no API keys for clients. The alpha is invitation-only and rate-limited by IP.
- **Monitoring/alerting stack** — No Prometheus, Grafana, or PagerDuty. Structured JSON logs (already in place from Sprint 4) are sufficient. Logs are tailed via `docker compose logs`.
- **CI/CD pipeline** — Manual `docker compose up` on the VPS. Automated deployments are a Sprint 6 concern.
- **Audio quality tuning** — Opus bitrate, FEC, DTX are set to sane defaults. Tuning is deferred until we have real user feedback.
- **Multi-room concurrent translation** — The architecture already supports it, but for alpha we cap at 5 simultaneous users (~2-3 rooms). No explicit multi-room optimization.

## Approach

### 1. Real Opus Codec (Hard Blocker #1)

The current `PassthroughCodec` in `internal/adapters/codec/opus_codec.go` forwards raw frames without conversion. The pipeline calls `codec.Decode()` (Opus -> PCM16) and `codec.Encode()` (PCM16 -> Opus) — these signatures are already correct in the `driven.AudioCodec` interface.

**Decision**: Use `github.com/pion/opus` — a pure-Go Opus decoder/encoder that requires no CGO and no system libraries. This is critical because:
1. The deployment target is a Linux container — CGO cross-compilation from Windows is painful.
2. The `hraban/opus` alternative requires `libopus-dev` installed at build time, adding container bloat.
3. `pion/opus` integrates naturally with the existing Pion ecosystem.

**Tradeoff**: `pion/opus` may have slightly lower performance than the C reference implementation. At 5 users this is irrelevant. If performance becomes an issue later, we swap to `hraban/opus` — the `driven.AudioCodec` interface isolates this decision completely.

**Implementation**: New `OpusCodec` struct in `internal/adapters/codec/` implementing `driven.AudioCodec`. The `PassthroughCodec` remains available for local development (build tag `//go:build !opus` or simply a constructor flag).

### 2. TURN Server Integration (Hard Blocker #2)

The current `webrtc.DefaultConfig()` returns STUN-only:
```go
ICEServers: []pionwebrtc.ICEServer{
    {URLs: []string{"stun:stun.l.google.com:19302"}},
}
```

~35-40% of users behind symmetric NAT cannot connect with STUN alone. TURN relays media through the server as a fallback.

**Decision**: Deploy Coturn as a Docker container alongside the Go server. The Go code changes are minimal:
1. Extend `webrtc.Config` to accept TURN URLs + credentials from environment variables.
2. Add the TURN server to the `ICEServers` slice in `DefaultConfig()` when env vars are present.
3. No new Go dependency — Pion already supports TURN natively via `pion/ice` and `pion/turn` (both already in `go.mod`).

**Coturn config**: Long-term credentials (static user/password), not TURN REST API. At 5 users, static credentials are fine. The credentials are shared between Coturn config and Go env vars.

**Tradeoff**: Static TURN credentials are less secure than short-lived tokens (TURN REST API). Acceptable for invitation-only alpha. Sprint 6 can add time-limited credentials if needed.

### 3. HTTPS/WSS via Caddy (Hard Blocker #3)

iOS and Android block plain WebSocket (`ws://`). Every modern browser marks non-HTTPS as insecure.

**Decision**: Caddy as reverse proxy in Docker Compose. Caddy handles:
- Automatic Let's Encrypt certificate provisioning and renewal
- HTTPS termination (port 443 -> Go server port 8080)
- WebSocket upgrade proxying (`/ws/*` routes)
- HTTP -> HTTPS redirect

The Go server requires ZERO TLS code changes. It keeps listening on `:8080` (plain HTTP) behind Caddy.

**Tradeoff**: Caddy adds ~30MB to the deployment. Alternatives (nginx, Traefik) require manual cert management. Caddy's automatic HTTPS is worth the tradeoff for a 5-user alpha.

**Prerequisite**: A real domain name pointing to the VPS IP. The proposal assumes the domain is configured before deployment.

### 4. Rate Limiting (Guardrail #1)

Each session calls OpenAI Realtime API. Without limits, a bad actor (or a bug) can generate unbounded cost.

**Decision**: Stdlib-only token bucket middleware. Two limits:
1. **Room creation**: max 3 rooms per IP per hour (prevents room-spam).
2. **WebSocket upgrade**: max 5 connections per IP per hour (prevents session-spam).

Implementation as an `http.Handler` middleware wrapping the existing `mux`. The bucket uses `sync.Mutex` + `time.Time` tracking — no external library. Configurable via `RATE_LIMIT_ROOMS_PER_HOUR` and `RATE_LIMIT_WS_PER_HOUR` env vars.

**Tradeoff**: IP-based limiting is coarse (shared IPs, NAT). At 5 invited users this is sufficient. If needed later, we add per-session or per-API-key limits.

### 5. Error UX (Guardrail #2)

Users currently get no actionable feedback when things go wrong.

**Decision**: Extend the existing `NotifySession` mechanism to send structured error events over WebSocket. The server already sends events like `room-closed` with a `reason` field. We add:

| Event Type | Trigger | Fields |
|---|---|---|
| `error:ice-failed` | ICE connection fails after timeout | `session_id`, `reason` |
| `error:server` | Internal error during pipeline | `session_id`, `stage`, `reason` |
| `error:rate-limited` | Client exceeds rate limit | `retry_after_seconds` |
| `error:translation` | OpenAI API error or timeout | `session_id`, `reason` |
| `error:codec` | Opus encode/decode failure | `session_id`, `stage`, `reason` |

The error events use the EXISTING signaling WebSocket — no new transport. The `EventNotifier` interface already supports arbitrary `msgType` + `fields` maps, so no interface change is needed.

**Tradeoff**: Error events are fire-and-forget (no delivery guarantee). If the WebSocket is down, the error is logged server-side only. Acceptable for alpha.

### 6. Feedback Endpoint (Guardrail #3)

**Decision**: Simplest possible — a `POST /feedback` handler in the HTTP server that:
1. Accepts `{"session_id": "...", "rating": 1-5, "comment": "..."}`.
2. Validates input (rating 1-5, comment max 500 chars).
3. Logs via `slog.Info("user_feedback", ...)` — structured JSON goes to stdout alongside all other logs.
4. Returns `201 Created`.

No database. No external service. Feedback is extracted from logs later using `jq` or any log aggregator.

**Tradeoff**: Feedback is mixed in with application logs. At 5 users with <50 feedback entries, this is fine. A dedicated store is a Sprint 6+ concern.

### 7. Deployment Topology

```
Internet
    |
    v
[Caddy :443] --TLS termination--> [TalkGo :8080]
    |                                    |
    |                              [Coturn :3478/:5349]
    |                                    ^
    +----TURN relay traffic--------------+
```

Single Docker Compose file. All three services on the same VPS. Internal Docker network for Go <-> Coturn communication.

Dockerfile for Go server: multi-stage build (Go 1.23 builder -> `gcr.io/distroless/static-debian12` runtime). Final image ~15MB.

### 8. Configuration Strategy

All configuration via environment variables, read in `cmd/server/main.go`:

| Variable | Default | Description |
|---|---|---|
| `LISTEN_ADDR` | `:8080` | HTTP listen address |
| `OPENAI_API_KEY` | (required) | OpenAI Realtime API key |
| `TURN_URLS` | (empty) | Comma-separated TURN URLs |
| `TURN_USERNAME` | (empty) | TURN static username |
| `TURN_CREDENTIAL` | (empty) | TURN static password |
| `RATE_LIMIT_ROOMS_PER_HOUR` | `10` | Max rooms per IP per hour |
| `RATE_LIMIT_WS_PER_HOUR` | `20` | Max WS upgrades per IP per hour |
| `GRACE_PERIOD` | `30s` | Room grace period on disconnect |
| `ROOM_TTL` | `10m` | Room inactivity TTL |
| `LOG_LEVEL` | `info` | Logging level |
| `CODEC_MODE` | `opus` | `opus` or `passthrough` |

## Risks

| Risk | Impact | Likelihood | Mitigation |
|---|---|---|---|
| `pion/opus` is immature or has bugs at 24kHz mono | Translation pipeline produces garbled audio | Medium | Write comprehensive encode/decode round-trip tests. Fallback: switch to `hraban/opus` with CGO (adds build complexity but is battle-tested). |
| Coturn misconfiguration blocks all TURN traffic | Users behind symmetric NAT can't connect | Medium | Test TURN with `turnutils_uclient` before deploying. Document config step-by-step. |
| OpenAI Realtime API costs exceed budget during alpha | Unexpected bill | Low (5 users) | Rate limiting caps sessions. Monitor via logs. Set OpenAI usage alerts. |
| Caddy fails to provision Let's Encrypt cert | No HTTPS = no mobile clients | Low | Pre-validate DNS. Caddy has clear error logs. Fallback: manual cert with `certbot`. |
| Single VPS goes down | Complete outage | Low | Alpha-acceptable. VPS provider has 99.9% SLA. Backup plan: redeploy on new VPS (Docker Compose makes this fast). |
| Rate limiter false-positives on shared IPs (corporate NAT, mobile carriers) | Legitimate users blocked | Medium | Set generous defaults. Make limits configurable. Log rate-limit events for debugging. |

## Definition of Done

All of the following must be true for Sprint 5 to be considered complete:

1. **Opus round-trip**: Audio encoded with `OpusCodec.Encode()` and decoded with `OpusCodec.Decode()` produces valid PCM16 output. Verified by unit test.
2. **Pipeline works end-to-end**: Two sessions in a room exchange audio through the full pipeline (capture -> decode -> translate -> encode -> send) with the real `OpusCodec`. Verified by integration test with mock translator.
3. **TURN connectivity**: A client behind simulated symmetric NAT connects successfully via the Coturn relay. Verified by `turnutils_uclient` test and/or manual test.
4. **HTTPS/WSS operational**: `https://<domain>/health` returns 200. WebSocket connects via `wss://<domain>/ws/<roomID>`. Verified by curl and manual test.
5. **Docker Compose deploys cleanly**: `docker compose up -d` on a fresh VPS starts all three services. `docker compose ps` shows all healthy. Verified by deployment doc walkthrough.
6. **Rate limiting active**: Exceeding `POST /rooms` or `GET /ws` limits returns HTTP 429 with `Retry-After`. Verified by unit test.
7. **Error events delivered**: At least `error:ice-failed`, `error:translation`, and `error:codec` events are sent to the client via WebSocket. Verified by unit tests with mocked ports.
8. **Feedback logged**: `POST /feedback` with valid payload returns 201 and produces a structured slog entry. Verified by unit test.
9. **All existing tests pass**: `make test` passes with zero failures. No regressions.
10. **Zero new Go dependencies except `pion/opus`**: `go.mod` diff shows only `pion/opus` added.
11. **Expo Go connects end-to-end**: A tester on iOS or Android running Expo Go can join a room and exchange audio with another peer via the production server. Verified by manual test with 2 devices.
